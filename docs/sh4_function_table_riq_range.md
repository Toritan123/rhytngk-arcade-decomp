# SH-4 function table — 0x0C070000 .. 0x0C071100 range

Generated by `tools/find_func_boundaries.py` against
`roms/fpr-24423_decrypted.bin`.  Method:
- Scan for prologue patterns (`sts.l pr,@-r15` / `mov.l rN,@-r15`).
- Walk each candidate forward until the first `rts` + delay slot.
- Collapse multi-prologue runs (e.g. functions saving r8, r14, then pr).
- Reject candidates that overshoot the next start by >32 bytes.

The full JSON lives in `build/sh4_functions.json` (9,213 functions
ROM-wide).  This document focuses on the range previously attributed to
`riq_title_init.c`.

## Gaps

There is a gap between functions #5 (ends 0x0C0701AC) and #6 (starts
0x0C0704F4) — roughly 840 bytes — where the scanner found no
prologues.  This region likely contains:
- Leaf functions with no callee-saved register usage (truly invisible to
  this scanner)
- Inline state-machine code with no formal function boundary
- Or static/const data embedded between code

The previously claimed fn0 (0x0C0701BA) and fn1 (0x0C070236)
would fall in this gap; without prologue evidence we cannot confirm
their existence or boundaries from this scan alone.

## Implications for src/riq_title/riq_title_init.c

The committed C file in `src/riq_title/riq_title_init.c` contains
fabricated function bodies whose addresses do not match the binary.
The file needs to be either:
- (a) Rewritten against the verified function table above, or
- (b) Re-attributed: most of these 42 functions probably do NOT belong
  to `riq_title_init.c` — the `riq_title/*.c` source split inside this
  address range is unknown.

Next step is to determine which subset of these 42 functions truly
belongs to `riq_title_init.c` vs sibling files (`riq_title_bsd.c`,
helpers, shared utility code).  Without `__FILE__` strings (stripped at
link time) the only reliable signal is call-graph clustering against
known-good entry points.
